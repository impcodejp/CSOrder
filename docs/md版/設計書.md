# CSOrder 設計書

最終更新: 2026-04-30

---

## 1. システム概要

CSOrder は **Tauri v2 + React (TypeScript) + SQLite** で構成されるデスクトップアプリケーションです。  
CS メンバーの受注案件を管理し、担当別・グループ別の売上・粗利・予算達成率を集計・出力することを目的としています。

### 技術スタック

| レイヤー | 技術 |
|---|---|
| フロントエンド | React 19 + TypeScript + Vite |
| バックエンド | Rust (Tauri v2) |
| DB | SQLite (sqlx 0.8 / 自動マイグレーション) |
| CSV デコード | encoding_rs (SJIS → UTF-8) |
| CSV パース | csv crate (serde Deserialize) |
| ファイルダイアログ | tauri-plugin-dialog |
| ファイル書き込み | tauri-plugin-fs |
| インストーラー | NSIS (Windows・日本語・全ユーザーインストール) |

### 3-5. 会計月番号の変換ルール

画面・CSV では暦月（4〜3月）を扱い、DB では会計月番号（1〜12）として保持する。

```
会計月番号 = ((暦月 - 4 + 12) % 12) + 1

例: 4月 → 1, 5月 → 2, 12月 → 9, 3月 → 12
逆変換: 暦月 = ((会計月番号 + 2) % 12) + 1
```

## 4. バックエンド設計

### 4-1. アーキテクチャ

```
フロント (invoke) → command → service → repository → SQLite
```

- **command**: 引数の受け取りと AppState の受け渡しのみ。ロジックを持たない。
- **service**: ビジネスロジック。repository を呼び出す。
- **repository**: SQL 実行のみ。ロジックを持たない。
- **model**: 各層で共有するデータ構造（struct）。

### 4-2. AppState

```rust
pub struct AppState {
    pub db: SqlitePool,   // SQLite コネクションプール（アプリ全体で共有）
}
```

起動時に `app_data_dir/csorder.db` を作成・接続し、マイグレーションを自動実行する。

### 4-4. service ロジック詳細

#### employee_service.rs

**`get_all_employees(pool)`**
- `employee_repository::get_all_employees_repo` を呼び、全社員を返す。

**`update_all_employees(pool, employees)`**
1. トランザクション開始
2. `employees` テーブルを全件 DELETE
3. 受け取った社員リストを INSERT
4. `budget` テーブルを全件 DELETE
5. `employees` から月次予算を展開し `budget` テーブルへ 12行/人 INSERT  
   （`monthly_budget1〜12` → `month_int 1〜12` の行に分解）
6. コミット

#### order_service.rs

**`get_all_orders(pool, cs_name, client_name)`**
- オプションのフィルター条件（担当者名・顧客名の部分一致）で絞り込み、受注一覧を返す。

#### report_service.rs

**`get_report(pool, from, to)`**

画面から受け取る `from`/`to` は暦月（例: 4, 5, 3）。  
DB の `order_month_int` は会計月番号のため、変換してから repository を呼ぶ。

```rust
let nfrom = ((from - 4 + 12) % 12) + 1;
let nto   = ((to   - 4 + 12) % 12) + 1;
report_repository::get_report(pool, nfrom, nto).await
```

**変換例:**

| 画面 (from/to) | DB (nfrom/nto) |
|---|---|
| 4 | 1 |
| 5 | 2 |
| 12 | 9 |
| 3 | 12 |

#### csv_import_service.rs

最も複雑なロジックを持つ。内部関数群 + 3つの公開関数で構成。

**内部関数**

| 関数 | 役割 |
|---|---|
| `read_sjis_file(path)` | ファイルをバイト列で読み込み、`encoding_rs` で SJIS → UTF-8 変換 |
| `validate_row(row, row_num)` | 1行の CsvRow を検証し `Ok(NewOrder)` または `Err(Vec<ValidationError>)` を返す |
| `parse_i64(value, field, row_num, errors)` | 文字列を i64 にパース。失敗時はエラーリストに追記して `None` を返す |
| `require_str(value, field, row_num, errors)` | 空文字チェック。空の場合はエラーリストに追記 |
| `date_to_month_int(date)` | `"YYYY/MM/DD"` → 会計月番号（`((月-4+12)%12)+1`）に変換 |
| `parse_csv(utf8)` | 全行をループし、契約日空欄行はスキップ、それ以外は `validate_row` で検証 |
| `errors_to_string(errors)` | エラーリストを `"行N: フィールド - メッセージ\n..."` 形式の文字列に変換 |

**`validate_row` の検証項目:**

| フィールド | 検証内容 |
|---|---|
| 案件番号 | 非空・i64 変換可能 |
| 案件主担当者ID | 非空・i64 変換可能 |
| 案件主担当者名 | 非空 |
| 案件名 | 非空 |
| 顧客名 | 非空 |
| 売上合計 | i64 変換可能 |
| 粗利合計 | i64 変換可能 |
| 契約日 | YYYY/MM/DD 形式・月が 1〜12 の範囲内 |

**公開関数**

| 関数 | 動作 |
|---|---|
| `preview_csv(path)` | `parse_csv` 実行。エラーがあれば `Err` を返す。なければ有効行を `Vec<CsvPreviewRow>` で返す |
| `validate_csv(path)` | `parse_csv` 実行。`CsvValidationResult`（合計/スキップ/有効数/エラーリスト）を返す |
| `import_orders(pool, path)` | `parse_csv` 実行。エラーがあれば `Err`。なければ全行を `upsert_orders` でDB登録し件数を返す |

**`parse_csv` のスキップ仕様:**
- `契約日` 列が空欄の行は **エラーではなくスキップ**（`skipped` カウントのみ加算）

**`upsert_orders` の動作:**
- `INSERT OR REPLACE INTO orders` を使用
- `project_code` の UNIQUE 制約により、既存レコードは削除→再挿入（実質上書き）

### 4-5. repository の SQL 詳細

#### report_repository::get_report

帳票の集計クエリ。`employees` をベースに `orders` と `budget` を LEFT JOIN する。

```sql
SELECT
    e.name        AS cs_name,
    e."group",
    e.grade,
    COALESCE(COUNT(o.id), 0)            AS project_count,
    COALESCE(SUM(o.amount), 0)          AS amount,
    COALESCE(SUM(o.gross_profit), 0)    AS gross_profit,
    COALESCE(b_sum.sum_budget, 0)       AS bugget
FROM employees e
-- 指定期間（nfrom〜nto）内の受注を結合
LEFT JOIN orders o
    ON  o.employee_code    = e.employee_code
    AND o.order_month_int BETWEEN ?2 AND ?3
-- 当月（nto）の予算（b_month は結合のみ、集計はb_sumで実施）
LEFT JOIN budget b_month
    ON  b_month.code      = e.employee_code
    AND b_month.month_int = ?1
-- 指定期間の予算合計
LEFT JOIN (
    SELECT code, SUM(budget) AS sum_budget
    FROM   budget
    WHERE  month_int BETWEEN ?2 AND ?3
    GROUP  BY code
) b_sum ON b_sum.code = e.employee_code
GROUP BY e.id, e.name, e."group", e.grade
ORDER BY e."group", e.grade DESC, e.name
```

バインドパラメータ: `?1 = nto`（当月）, `?2 = nfrom`（期間開始）, `?3 = nto`（期間終了）

> ※ 返却されるのは **指定期間（from〜to月）の合計** 1セット。フロントエンドで当月と累計を分けた表示はしない。  
> 帳票フロントは `from == to` の場合に「4月件数」、異なる場合に「4~6月件数」のようにラベルを切り替える。

```
[プレビューテーブル表示]（担当者名・売上・粗利・案件名・顧客名・契約日）
    ↓ ユーザーが確認
[インポート実行]
    ↓ invoke("import_orders", { path })
[完了メッセージ（取込件数）]
```

**エラー時:** catch ブロックでエラーメッセージを表示  
**インポートボタン:** ファイル選択後に表示、プレビュー0件の場合は disabled

### 6-4. 帳票 (Report.tsx)

**機能:** 担当別・グループ集計の2タブ表示 + CSV出力  
**フィルター:** 期間（from月〜to月）を入力し「更新」で再取得  
**取得:** `get_report(from, to)`（`appliedFrom`/`appliedTo` 変更時に `useEffect` で自動実行）

**担当別タブの表示列（from == to の場合）:**

| 列 | 内容 |
|---|---|
| 名前 / GROUP / GRADE | 社員情報 |
| {to}月件数 | 受注件数 |
| {to}月売上 | 売上合計（千円） |
| {to}月粗利 | 粗利合計（千円） |
| {to}月予算 | 予算合計（千円） |
| {to}月達成率 | 粗利 ÷ 予算 × 100（%） |

> from ≠ to の場合は「{from}~{to}月〇〇」と表示。

**グループ小計行:** 各グループの最終行に小計行を挿入（`subtotalRow` スタイル）  
**合計行:** tfoot に全体合計を表示  
**達成率の色分け:** ≥100% → 緑系（`rateHigh`）、<80% → 赤系（`rateLow`）

## 8. エラーハンドリング方針

| レイヤー | 方針 |
|---|---|
| repository | sqlx エラーを `.to_string()` で `Err(String)` に変換して上位へ伝播 |
| service | repository の `Err` をそのまま伝播。ビジネスルール違反も `Err(String)` |
| command | `Result<T, String>` をそのまま Tauri に返す。Tauri が JSON シリアライズ |
| フロントエンド | `invoke` の catch ブロックで `String(err)` を画面表示 |
